# 일기당천 2.5D 게임 구현 가이드

이 문서는 “Three.js로 그림을 띄우는 법”이 아니라, 2.5D 전장을 **읽히고 반복 플레이할 수 있는 게임**으로 만드는 기준이다.

## 1. 전장의 역할

배경 오브젝트는 아래 셋 중 하나의 역할을 가져야 한다.

1. **랜드마크** — 성문·망루·봉화대가 현재 지역과 진행 방향을 알려준다.
2. **전투 공간** — 낮은 목책·잔해가 안전한 쪽과 위험한 쪽을 만든다.
3. **전투 선택** — 화약 수레, 만두 수레, 사당이 공격·회복·버프 선택을 만든다.

역할 없는 장식은 화면 가장자리와 원경에만 둔다. 전장은 30m 청크의 방/안뜰과 폭 7m 성문으로
연결한다. 각 청크는 막다른 길·직선 통로·모서리·삼거리·교차로 중 하나가 되며, 모든 청크는
원점까지 이어지는 spanning tree와 일부 순환로를 공유한다. 방 안은 대군 전투가 가능하게 넓게 두되,
다음 방으로 이동하려면 실제 성문이나 갈림길을 통과해야 한다.

## 2. 2.5D 렌더 규칙

- 카메라는 고정 55도 원근 카메라를 유지한다.
- 캐릭터와 월드 오브젝트는 발(bottom-center)을 원점으로 한 기울어진 평면을 쓴다.
- 알파 컷아웃은 `discard` 후 불투명으로 그려 깊이 버퍼를 기록한다. 반투명 정렬에 의존하지 않는다.
- 월드킷 12종은 512×512 한 장의 아틀라스와 InstancedMesh 배치로 그린다.
- 호로관은 128px 소품 셀에 넣지 않고 1024×512 독립 랜드마크로 그린다.
- 연결 성벽도 호로관과 같은 화풍의 256px 모듈(직선/측벽/망루/문루/균열/붕괴)을 실제 collider 길이에 반복 배치한다.
- 성벽 footprint는 월드 좌표 기반 픽셀 석재 셰이더의 3D Box InstancedMesh로 두께를 만들고,
  같은 위치에 도트 파사드를 겹친다.
- `NearestFilter`를 확대 필터로 유지하고, 축소 시에는 nearest mipmap을 써서 원거리 깜빡임을 줄인다.
- 색 텍스처는 sRGB로 지정하고 셰이더에서 감마를 중복 적용하지 않는다.
- 정적 랜드마크와 동적 기믹은 배치를 나눈다. 텍스처는 공유한다.

## 3. 전투 패턴 문법

일반 적의 특수 행동은 복잡한 행동 트리 대신 작은 상태 머신으로 통일한다.

```text
대기/추적 → 예고(Windup) → 공격(Action) → 빈틈(Recovery) → 대기/추적
```

| 적 | 예고 | 공격 | 빈틈과 대응 |
|---|---|---|---|
| 돌격병/창병 | 0.68초 붉은 원, 방향 고정 | 0.42초 직선 돌진 | 0.95초 감속. 옆으로 피한 뒤 공격 |
| 궁병 | 0.72초 황색 원 | 3발 부채꼴 화살 | 0.60초 정지. 틈에 파고들기 |
| 책사 | 0.92초 보라 원 | 3발 유도 마탄 | 0.85초 정지. 이동 지속 또는 먼저 처치 |
| 방패병 | 평상시 피해 45% 감소 | 3타를 받으면 방패 파괴 | 2.4초 그로기. 집중 공격 보상 |

동시에 예고/공격하는 특수 패턴은 최대 3개로 제한한다. 나머지 적은 기본 추적을 유지해
대군의 밀도는 살리되, 피할 수 없는 패턴 중첩을 막는다.

## 4. 10분 페이싱

한 구간에서 새 문법은 하나만 가르치고, 다음 구간에서 조합한다.

| 시간 | 학습/조합 목표 |
|---|---|
| 0:00–1:00 | 기본 추적 + 일부 척후 돌격. 이동과 회피 학습 |
| 1:00–2:00 | 방패병. 같은 대상을 여러 번 때리는 보상 학습 |
| 2:00–3:00 | 창병 돌진. 예고와 빈틈 학습 |
| 3:00–4:30 | 원소 보스. 일반 특수 패턴 생성은 줄여 보스에 집중 |
| 4:30–6:00 | 궁병 3연사 + 포위. 원거리 우선순위 학습 |
| 6:00–7:30 | 동탁 + 책사. 보스와 유도탄을 겹치되 동시 특수 패턴 수 제한 |
| 7:30–9:00 | 모든 문법 조합, 보급/화약/목책으로 회복 구간 제공 |
| 9:00–10:00 | 여포. 엘리트와 포위 이벤트를 멈추고 최종보스에 집중 |

엘리트·포위 이벤트는 보스 중 일시 중지한다. 보스 등장 프레임에도 스포너가 먼저 실행되지 않도록
보스 활성 상태를 스포너에 먼저 전달한다.

## 5. 구조물과 전투의 연결

- **성문**: 돌격병이 밀려오는 방향을 시각적으로 설명한다.
- **망루/북대**: 궁병 일제사격과 경고색을 연결한다.
- **책사 진영/사당**: 보라색 마탄과 버프의 출처를 설명한다.
- **화약 수레**: 공격적인 위치 선정을 광역 처치로 보상한다.
- **목책**: 부수면 작은 충격파가 발생해 전선을 열어 준다.
- **만두 수레**: 후반의 짧은 회복 박자를 만든다.

큰 성벽은 실제 circle-vs-AABB 충돌을 사용한다. 플레이어는 축 분리 슬라이드로 벽면을 따라 움직이고,
적은 플레이어 주변 96m의 1.5m 격자 flow field를 0.22초마다 갱신해 문을 찾아간다. 스폰 지점도
막힌 셀에서 가장 가까운 보행 가능 위치로 투영한다. 이 세 규칙을 함께 넣지 않고 플레이어 충돌만
추가하면 대군이 벽 반대편에 쌓이므로 맵 기능은 완성된 것으로 보지 않는다.

## 6. 호로관 파성 와우 모먼트

원점 북문은 닫힌 대형 호로관으로 시작한다. 문 앞 10m 안에서 적 12명을 처치하면:

1. 성문 collider와 대형 파사드가 제거되고 무너진 성벽 스프라이트로 교체된다.
2. 48개(모바일 28개) 3D 픽셀 석재/목재 파편이 Y축으로 솟으며 깊이 버퍼 앞뒤를 가른다.
3. 먹빛 충격파 2겹, 먼지 72개, 90ms hitstop, 카메라 trauma와 FOV 펀치가 겹친다.
4. 0.8초 뒤 8 돌격병 + 2 창병이 열린 길 안쪽에서 쏟아진다.
5. 플레이어와 적 모두 이전 성문 위치를 실제로 통과할 수 있다.

평상시 2.5D 카드가 파성 순간 실제 3D 파편으로 전환되는 대비가 Three.js 사용 이유를 보여준다.

## 7. 성능 예산

| 항목 | 데스크톱 | 모바일 |
|---|---:|---:|
| draw calls | 120 이하 | 100 이하 |
| triangles | 200k 이하 | 100k 이하 |
| textures | 96 이하 | 96 이하 |
| geometries | 64 이하 | 64 이하 |
| 월드킷 추가 비용 | draw call 2, 텍스처 1 공유 | 동일 |

정적 월드는 플레이어가 30m 청크 경계를 넘을 때만 재배치한다. 충돌은 5×5 청크, 렌더는 3×3 청크를
유지한다. 적 근접 질의는 기존 2m 공간 해시, 벽 내비게이션은 별도 flow field를 사용한다.

## 8. 완료 판정

- 첫 전투 화면에서 성문/성벽 계열 랜드마크와 두 종류 이상의 소품이 보인다.
- WASD로 벽을 관통하지 못하고 벽면을 따라 미끄러지며, 문을 통해 다음 청크로 이동한다.
- 적이 벽 안에서 생성되지 않고 flow field를 따라 문으로 유입된다.
- 호로관 파성 전후에 collider가 1개 제거되고, 파편이 보이며, 이전 문 위치를 통과한다.
- 돌격·궁병·책사 패턴 각각이 예고 후 공격하고, 패턴 카운터가 증가한다.
- 보스 중 엘리트/포위가 추가로 겹치지 않는다.
- 데스크톱과 모바일에서 콘솔 오류가 없고 위 성능 예산 안에 든다.
- 생성 에셋은 기존 캐릭터처럼 픽셀 가장자리가 선명하고 검은 판처럼 뭉개지지 않는다.
